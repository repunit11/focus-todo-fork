# AGENTS

## TDD運用ルール
- すべての実装は Red -> Green -> Refactor の順で進める。
- Red では失敗するテストを先に追加し、期待する振る舞いをテスト名とアサーションで固定する。
- Green ではテストを通す最小実装のみを行う。
- Refactor では振る舞いを変えずに設計改善を行い、対象テストを再実行して全緑を確認する。
- 機能追加時はユニットテストを優先し、境界をまたぐ振る舞いは統合テストで補う。

## コミット運用ルール
- コミットは Red / Green / Refactor ごとに分ける（1コミット1責務）。
- コミットメッセージは `test|feat|refactor(<scope>): <summary>` を使用する。
- 変更対象に無関係な差分や自動生成物を同一コミットに含めない。
- コミット前に対象テストを実行し、結果を確認する。
- ユーザーから都度指定がなくても、小さい単位でのコミットをデフォルト運用とする。
- 各作業ターンの完了時に、そのターンで作成したコミットID一覧を共有する。

## 各レイヤーのDoD
- Domain: ユースケースの入出力と失敗条件がテストで定義されている。
- Infrastructure: DBアクセスや外部I/Fの失敗系を含むテストがある。
- API: 正常系と代表的な異常系のレスポンス仕様がテストで固定されている。
- Frontend: 主要操作フローに対するUIテストまたはE2Eテストがある。

## 禁止事項
- テスト未追加のまま仕様変更を実装しない。
- 大きなまとめコミットを作らない。
- Refactorコミットで仕様変更を混ぜない。
- 失敗テストを放置したまま次機能に進まない。
